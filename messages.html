I will focus on electrum plugin because I have not updated other clients since long time

2 ways to start the coinjoin process

1. Create a pool
2. View and Join existing pools

- If a user creates a pool it is a public nostr event that shares info like id, denomination, number of peers, timeout, relay, public key
- This is followed by pool sending a message to self sharing a new address
- This information can be used by others to join
- If a user tries to join a pool it sends an encrypted message to public key of pool
- Pool (user who created it) responds with credentials
- Other user joins the pool by sharing a new address
- When all peers are done sharing their addresses, they are asked to select one input and share it which automatically gets signed using sighash all | acp
- Once all inputs are shared in pool, a transaction is created and broadcasted

Following up (thanks for the link from delving) .. it's definitely not obvious that the autct style method actually is helpful, it depends on .. stuff... The outline of "why might this thing be useful?" is probably in the first few sections of https://reyify.com/blog/riddle . Basically, "paying" to take part in a protocol without violating privacy. You can try to do that in other ways, most obviously a Lightning payment, but that might be suboptimal privacy wise, as well as requiring more baggage/interaction. Ring sigs are another direction (which I've pursued in multiple blogs and code examples last two years), but they limit you in anon set to say 10**3 kind of anon set size. Curve trees lets you got 10**6 if you want.
12:38
Note also that "payment" is in scare quotes because of course there's no literal payment, which means the "cost" is very minor; it only will really provide anti-sybil at large scales/sizes. so if you really care if there are say 3 participants in your protocol and 2 of them are sybils of the first, this barely helps if at all. It's more for stopping a large scale "overwhelming" attack like e.g. being 99 of 100 in a coinjoin and doing that repeatedly. In this regard it's very similar to fidelity bonds as currently implemented in Joinmarket, but with a different tradeoff: fidelity bonds currently are completely exposed as in the utxo itself is published, on the other hand, the use of long time locks at least means the cost per individual user is non-trivial (time value of money). On the other extreme, autct tokens do not require you to lock up the funds, so even if you use a high threshold like 1M sats or 10M sats it's still a pretty minor cost (tx fee, and having access to funds rather than spending them), but the advantage is the anon set is extremely high and there are no on chain actions so its privacy is about as good as it gets.
12:40
(A final technical note: if you read the curve trees paper what we are looking at here is "zero knowledge set membership" proof, but that in itself doesn't get anonymous usage tokens - you'll see an extra pdf in my aut-ct repo that explains how you can "attach" a token (key image) to that proof. It's pretty straightforward but I did double check with one of the paper authors and they agreed with my approach on that)

from PyQt5.QtCore import QObject, pyqtSignal, QThread

class Worker(QObject):
    changeParentRequest = pyqtSignal(QObject)

    def __init__(self, child):
        super().__init__()
        self.child = child

    def run(self):
        # This is called in a different thread
        # Emit signal to request parent change
        self.changeParentRequest.emit(self.child)

class MainObject(QObject):
    def __init__(self, parent=None):
        super().__init__(parent)
        self.thread = QThread()
        self.worker = Worker(self)
        self.worker.moveToThread(self.thread)
        self.worker.changeParentRequest.connect(self.changeParent)
        self.thread.started.connect(self.worker.run)
        self.thread.start()

    def changeParent(self, child):
        # This slot will be executed in the main thread
        child.setParent(self)

if __name__ == "__main__":

Each user generates a cryptographic key pair.
They then create a "commitment" to an output using a secret value. This is like saying, "I have a secret that I promise to reveal later."

All commitments are added to a special cryptographic structure called an "accumulator." This helps us keep track of all commitments without revealing them. it also lets us prune any outputs that don't have an input later registered.

When users later register their inputs, they need to prove they know the secret related to their commitment without actually revealing the secret or linking it to their output.

Any outputs that don't have a ZK proof later provided are pruned.

I found a solution for this:

- - Redo output registration if a participant does not respond and fails to register inputs.
- - Keep a minimum number of peers for pool.
- - Use ring signatures for re-attempt because in this case we already have inputs and need outputs.
- - Coinjoin works even if some participants do not register inputs.

Example: Alice, Bob and Carol in the pool with output already registered. Carol fails to register input. Alice and Bob register new addresses as output after N seconds with a proof that one of the registered input belongs to them. New tx is created and signed.
